01
Phase One

Build Ground Truth First

Before you search for anything, build a machine that can judge correctness. Verification comes before exploration.

The first question is never "how do I solve this?" It is: "how will I know when I have solved it?" Define the success condition precisely in code before writing a single line of search. Without this, you are exploring blind.

⚖️
Verification Before Search

Write verify() before write search(). A fast checker is a force multiplier for everything that follows.

🔬
Measure the Object

Count vertices, edges, degrees. Draw small examples by hand. Know the exact shape of what you are dealing with before touching it.

🪞
Test the Verifier

Deliberately feed your checker a wrong answer. If it does not catch it, your ground truth is not true.

# This was the actual first script. # No search. No solution attempt. Just: can I even check an answer? def verify_sigma(sigma, m): funcs = build_functional_graphs(sigma, m) for c, fg in enumerate(funcs): if not is_single_hamiltonian_cycle(fg, m): → return False, f"cycle {c} is not Hamiltonian" → return True, "valid decomposition" # Now I can search. Without this: searching in the dark.
PRINCIPLE 1.1
"The verifier is worth more than the solver."
A perfect solver without a verifier produces untrustworthy results. A perfect verifier with a random search produces verified results. Build the verifier first, always.
02
Phase Two

Hit the Wall Deliberately

Brute force is not a failure. It is a measurement instrument. The way it fails tells you exactly what structure you need to find.

The search space for m=3 is 6²⁷ ≈ 150 trillion. Too slow. But how it was too slow was informative: the bottleneck was not the solver — it was that we had no constraint to prune on. The absence of fast pruning meant we lacked a compact representation of what makes an assignment valid.

✗
Linear formula σ(i,j,k) = (a·i + b·j + c·k) mod 3
Failed for all (a,b,c). Lesson: the solution is not linear. We need non-uniform assignment — it must depend on position in a structured way.
✗
DFS over all 6²⁷ sigma assignments (m=3)
Too slow. Lesson: 27 vertices × 6 choices each is too large without pruning. We need a constraint that eliminates most branches early.
✗
Cyclic rotation σ(i,j,k) = rotate_by(f(j,s))
Produced in-degree violations. Lesson: sigma must ensure each cycle is bijective at every level — this is the real constraint, not just assignment.
→
The failures converged on a common signal
Key insight: every failure had the same shape — in-degree violations. The problem was always about bijections, not permutations. This pointed toward a layered structure.
PRINCIPLE 2.1
"Failures are diagnostic, not terminal."
Each failed approach narrows the solution space and reveals the hidden structure. Catalog failures explicitly — the pattern across failures is usually more important than any individual attempt.
❌ Thinking before failures

"I need to find a sigma that works. Try all sigmas."

✓ Thinking after failures

"The failures all involve bijection. The problem has layer structure. I need to decompose it level by level."

03
Phase Three

Hunt for the Hidden Layer

Every hard combinatorial problem has a natural decomposition that makes it tractable. The search for structure is more important than the search for solutions.

The question that unlocked everything was not "what is the answer?" but "what is the natural structure of this object?" The digraph had a natural stratification via f(i,j,k) = i+j+k mod m. Once seen, it reduced the 3D problem to a sequence of 2D bijection problems — and those were solvable.

🔭
Ask: What is the Natural Symmetry?

Every well-posed combinatorial object has a symmetry group. Find it. It almost always reveals the right decomposition.

📐
Reduce Dimensions

Can you project from 3D to 2D? From 2D to 1D? Each dimension reduction multiplies the tractability of search.

🧅
Find the Layering

If the problem resists direct solution, look for a function that stratifies it into levels where each level is independently solvable.

🔑
Name the Key Constraint

The bijection condition — "at each level, each cycle hits every fiber position exactly once" — was the constraint that made backtracking efficient. Find the constraint that prunes hardest.

# The discovery: f(i,j,k) = i+j+k mod m stratifies the digraph. # Every arc goes from fiber F_s to F_{s+1}. # The 3D problem becomes: m independent 2D bijection problems. Fiber F_s = {(i,j,k) : i+j+k ≡ s mod m} # m² vertices each Arc types in fiber coordinates (i,j): arc 0 → shift (1,0) # increment i arc 1 → shift (0,1) # increment j arc 2 → shift (0,0) # identity ← THIS is the key insight # Now the bijection constraint is LOCAL: at each level s, column j, # the 3 arcs go to 3 cycles such that each cycle's targets span all of Z_m². # This reduces 6^{m³} search space to (valid_levels)^m — exponentially smaller.
PRINCIPLE 3.1
"The right decomposition makes the hard part obvious."
When you find the correct structure, the difficulty collapses. The problem does not become easier — your view of it changes so that the difficulty was always in finding the frame, not the solution.
04
Phase Four

Reverse-Engineer a Working Solution

Once you have any valid answer, stop and study it before moving on. The solution knows something you do not. Extract that knowledge.

The backtracking search found a valid sigma for m=3. The instinct was to immediately try m=4. The right move was the opposite: stop, read the solution, and ask what pattern it is instantiating. The m=3 solution contained the general theorem — but only if you looked.

🔍
Read Solutions as Theorems

A specific valid answer is a compressed proof of existence. Decompress it. What principle is it demonstrating?

📊
Tabulate, Don't Stare

Print the solution in every coordinate system you can think of. The pattern appears in one of them.

📉
Ask: What is the Minimal Representation?

Does the solution depend on all three coordinates, or just two? Just one? Minimizing dependencies reveals the essential structure.

# m=3 solution was found. First question: what does it depend on? sigma(i=0, j=1, k=2) → [1, 0, 2] sigma(i=1, j=1, k=1) → [1, 0, 2] # same j, same s=(i+j+k)%m=0 sigma(i=2, j=1, k=0) → [1, 0, 2] # same j, same s=0 # Pattern detected: sigma depends only on (s, j), NOT on i separately. # Column-uniformity. This reduces the sigma space from m³ points to m² points. # Next: compute Q_c = composition of all m fiber-level functions. # Found: Q_c(i,j) = (i + b_c(j), j + r_c) mod m # "Twisted translation" — uniform j-shift, j-dependent i-shift.
PRINCIPLE 4.1
"One working example contains the general law."
In mathematics and computation, a specific solution is almost always an instance of a general pattern. The habit of reading solutions analytically — rather than just recording them — is the core habit of discovery.
❌ Common mistake

"m=3 works. On to m=4. And m=5. Collect solutions."

✓ Discovery move

"m=3 works. Why does it work? What law does it instantiate? That law will solve m=5,7,9,... simultaneously."

05
Phase Five

Lift to the General Case

A pattern in one case is a hypothesis. Verify it in two more cases, then state it as a theorem and find the condition it requires.

The twisted translation form Q_c(i,j) = (i + b_c(j), j + r_c) appeared in m=3 and m=5. The question then became: what condition on b_c and r_c makes Q_c a single m²-cycle? And when can those conditions be satisfied? These two questions — one analytical, one existential — gave us the complete theorem.

🧪
Three Cases Verify a Pattern

One case is a coincidence. Two cases suggest a rule. Three cases with the same structure confirm you have found something real.

❓
Find the Governing Condition

Ask: what is the exact condition on the parameters that makes this work? The condition is the theorem. The specific values are just examples.

🌐
Parametrize the Family

Replace specific values with variables. Replace "m=3" with "odd m". Replace "[1,2,2]" with "b_c(j)". The general statement is always more valuable than the instance.

# The governing condition for Q_c to be a single m²-Hamiltonian cycle: Condition A: gcd(r_c, m) = 1 # j-shift coprime to m Condition B: gcd(Σⱼ b_c(j), m) = 1 # total i-shift coprime to m # Verified for m=3,5,7. Now: when can both conditions be satisfied? # For odd m: need r₀+r₁+r₂ = m, each r_c coprime to m. # Solution: (r₀,r₁,r₂) = (1, m-2, 1) always works when m is odd. # gcd(1, m) = 1 ✓ # gcd(m-2, m) = gcd(2, m) = 1 for odd m ✓ # Sum = m ✓ # THEOREM: For all odd m > 2, the construction exists. ✅
PRINCIPLE 5.1
"The condition is more valuable than the example."
Finding that m=3 works is a data point. Finding the condition gcd(r_c,m)=1 is a theorem that covers m=3,5,7,9,11,... simultaneously. Always seek the condition, not the list of cases.
06
Phase Six

Find What Cannot Work

An impossibility proof is not a defeat. It is the most precise possible description of the boundary of the solution space — and that boundary is exactly where understanding lives.

m=4 resisted every approach. The fiber-based search found nothing in 5 million configurations. The instinct was to try harder. The right move was to ask why it cannot work. The answer — three odd numbers cannot sum to an even number — is a three-line proof that eliminates half the integers and reveals why even m is fundamentally different.

🚧
Resistance is a Signal

When search consistently fails and you have verified the implementation, the problem likely has no solution in that class. Switch from search to proof.

🎯
Compute the Obstruction

Ask: what would have to be true for a solution to exist? Is that thing possible? Usually the obstruction is a parity argument, a modular constraint, or a counting argument.

↔️
Impossibility Proves the Boundary

Showing odd m always works AND even m (column-uniform) never works draws a precise boundary. That boundary is a theorem. It is more valuable than either half alone.

# The impossibility proof for even m (column-uniform approach): Step 1: Q_c needs gcd(r_c, m) = 1 # single-cycle condition Step 2: For even m, gcd(r, m) = 1 ⟹ r is odd (coprime to even = no shared factor 2 = odd) Step 3: We need r₀ + r₁ + r₂ = m Step 4: Sum of 3 odd numbers = odd Step 5: But m is even. odd ≠ even. → Contradiction. ∎ # Three lines. Covers ALL even m simultaneously. # Tells us: even m requires a fundamentally different construction. # The search was not failing due to bugs — it was hitting a real wall.
PRINCIPLE 6.1
"What cannot work is as important as what can."
The impossibility result did not close the problem — it clarified it. It revealed that even m requires a different mechanism, which is a deeper understanding than any specific solution provides.
Synthesis

The Core Laws of Discovery

Distilled from the full arc. These apply regardless of domain — mathematics, engineering, science, software.

LAW I
"Build the judge before the player."
A verifier that can evaluate any candidate solution is worth more than a hundred search heuristics. It makes every subsequent step honest.
LAW II
"Measure how you fail, not just that you failed."
The structure of failure is data. Catalog what went wrong, at what scale, with what symptom. The common thread across failures points to the missing structure.
LAW III
"The right representation makes the problem trivial."
The fiber decomposition turned a 3D problem with 6²⁷ configurations into a sequence of 2D problems with 48⁴ configurations. The mathematics did not change — the view changed.
LAW IV
"A solution is a compressed theorem. Decompress it."
Reading any working answer analytically — asking why it works, what it depends on, what its minimal representation is — almost always reveals the general law hiding inside the specific case.
LAW V
"Name the condition, not the cases."
"Works for m=3,5,7" is a list. "Works for all odd m because gcd(r_c,m)=1 can always be satisfied" is a theorem. The condition is infinitely more powerful than the enumeration.
LAW VI
"Resistance demands proof, not persistence."
When a systematic search repeatedly fails across different implementations and seeds, stop searching and start proving. The wall is usually real and has a name.
Applied Universally

Universal Operating Rules

These are the actionable habits. They apply to any domain: software engineering, scientific research, mathematical problem solving, product design.

R1
Verify Before You Claim
Every claim in a discovery process must be backed by a test that could have failed. Running a test that always passes proves nothing.
R2
Record Every Failure Precisely
What was the input? What was the symptom? What did it rule out? A failure without a record is wasted information.
R3
Ask "What Does This Depend On?"
For every solution or pattern found, reduce its dependencies to the minimum. Fewer dependencies = more generality.
R4
Change the Representation
When stuck, do not try harder — look for a different coordinate system, a different encoding, a different level of abstraction. Most breakthroughs are re-framings.
R5
Pause After Each Find
Every successful result deserves 5 minutes of reading before the next search begins. The solution contains the next hypothesis.
R6
State the Impossibility
Every construction theorem should be paired with an impossibility result. Knowing where the construction fails proves you understand it.
R7
Work Level by Level
When a problem resists direct attack, find the natural stratification. Solve one level at a time with local constraints. Compose the global solution from local bijections.
R8
Name What You Found
Give every discovered structure a precise name and formula. "Twisted translation" is more useful than "that pattern in the Q table." Naming crystallizes understanding.

Origin

These principles were extracted by analyzing the step-by-step process used to solve Knuth's "Claude's Cycles" problem — a February 2026 open problem about decomposing a directed Cayley graph into Hamiltonian cycles.

The specific problem is combinatorics. The methodology is universal.

How to Use This

  • Apply the Discovery Arc to any hard problem you are stuck on.
  • Use the six phases as a checklist: are you really in Phase 3 or still in Phase 2?
  • The Laws are the most important part — return to them when progress stalls.
  • The Universal Rules are daily habits, not one-time steps.
